    Asynchronous JavaScript – Event Loop, Web APIs, setTimeout & Callback Hell

    JavaScript Nature (VERY IMPORTANT)

    Golden Rule:
    JavaScript is ALWAYS synchronous.

    Even if it behaves asynchronously, the core nature does not change.

    👉 From class: 

    
JavaScript = Single-threaded + Synchronous

    As explained in the session, async behavior comes from external systems like Web APIs, not JavaScript itself. :contentReference[oaicite:0]{index=0}

    ---

    What is Synchronous?

    Synchronous means execution happens step by step in order. 
    
console.log("Start");
console.log("Wait for 2 seconds");
console.log("End");

    Output

    
Start
Wait for 2 seconds
End

    👉 JavaScript does NOT wait. It just executes line by line instantly.

    ---

    Why JavaScript Feels Asynchronous?

    Because of Web APIs.

    Web APIs = Superpowers of the browser (NOT JavaScript)

    Examples

    • setTimeout
    • setInterval
    • fetch
    • console
    • localStorage

    👉 Even console.log is NOT part of JavaScript. :contentReference[oaicite:1]{index=1}

    ---

    setTimeout Explained

    
console.log("Start");

setTimeout(() => {
  console.log("Waited 5 seconds");
}, 5000);

console.log("End");

    Output

    
Start
End
Waited 5 seconds

    Why?

    • JavaScript executes synchronously
    • setTimeout is handled by browser
    • JS does NOT wait
    ---

    Behind the Scenes (MOST IMPORTANT)

    Components

    • Call Stack (JS Engine)
    • Web APIs (Browser)
    • Callback Queue
    • Microtask Queue
    • Event Loop
    ---

    Execution Flow

    1. Code enters Call Stack
    2. setTimeout is sent to Web APIs
    3. Timer starts in browser
    4. Callback stored in memory
    5. After timer → goes to Callback Queue
    6. Event Loop checks if Call Stack is empty
    7. If empty → moves callback to Call Stack
    8. Callback executes
    ---

    Event Loop (Core Concept)

    Event Loop continuously checks:
    • Is Call Stack empty?
    • If yes → take task from Queue

    👉 It runs continuously like a loop.

    ---

    Multiple setTimeout Example

    
setTimeout(() => console.log("2 sec"), 2000);
setTimeout(() => console.log("3 sec"), 3000);

    Key Insight

    • Timers run in parallel
    • Total time = max delay

    👉 Total time = 3 seconds (not 5) :contentReference[oaicite:2]{index=2}

    ---

    setInterval (Repeating Tasks)

    
setInterval(() => {
  console.log("Running...");
}, 2000);

    👉 Runs every 2 seconds infinitely

    ---

    Stopping Interval

    
let id = setInterval(() => {
  console.log("Running...");
}, 2000);

setTimeout(() => {
  clearInterval(id);
}, 10000);

    👉 Stops after 10 seconds

    ---

    Callback Hell (Pyramid of Doom)

    Problem Statement

    Perform dependent tasks:
    • 1. Select Image (4s)
    • 2. Apply Filter (2s)
    • 3. Add Caption (5s)
    • 4. Upload (3s)
    ---

    Solution (Nested Callbacks)

    
setTimeout(() => {
  console.log("Image Selected");

  setTimeout(() => {
    console.log("Filter Applied");

    setTimeout(() => {
      console.log("Caption Added");

      setTimeout(() => {
        console.log("Image Uploaded");
      }, 3000);

    }, 5000);

  }, 2000);

}, 4000);
    ---

    Problem

    This structure is called: Callback Hell / Pyramid of Doom
    ---

    Why It’s Bad?

    • Hard to read
    • Hard to debug
    • Hard to handle errors
    • Code grows horizontally

    👉 From session:

    "Error handling becomes difficult in callback hell" :contentReference[oaicite:3]{index=3}

    Better Solutions

    • Callbacks ❌
    • Promises ✅
    • Async/Await 🚀 (Best)

JavaScript → synchronous
Browser → handles async
Event Loop → manages execution
